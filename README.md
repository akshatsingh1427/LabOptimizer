## What This Project Does

This is a two-part system:

1. **`lab_system.cpp`** — an interactive C++ CLI that takes student/lab/subject data (typed in or piped via a file like `input.txt`), runs 8 algorithms over it (scheduling, allocation, cheating detection, critical-lab detection, etc.), times every algorithm with `chrono`, and writes a full JSON report to `output.txt`.
2. **`dashboard.html`** — a browser dashboard with 10 tabs that loads `output.txt` (via drag-and-drop or file picker) and turns the raw JSON into charts, bar graphs, an allocation animation, a heatmap, a "what-if" simulator, and algorithm explainer cards.

`lab_system.exe` is the compiled Windows binary of `lab_system.cpp`.

## How the Pieces Connect

```
1. Run lab_system.exe (or a compiled lab_system.cpp)
2. Enter/pipe in: student count, lab count, subject count, dependency count,
   student records (id, subject, branch), subject prerequisites, lab capacities
   → this is saved back out as input.txt
3. The program runs all 8 algorithms with real chrono timing
   → results are written to output.txt as JSON
4. Open dashboard.html in a browser
5. Drag output.txt onto the drop zone (or use the file picker)
   → dashboard populates all 10 tabs from that JSON
```

## Input Format (`input.txt`)

```
N L S D                          ← students, labs, subjects, dependencies
<N lines>  student_id subject_id branch_id
<D lines>  subjectA subjectB     ← subjectA must complete before subjectB
<1 line>   L lab capacities, space-separated
```

The sample `input.txt` here: 100 students, 8 labs, 10 subjects, 15 dependency edges, and lab capacities `6 8 20 45 31 12 19 9`.

## The 8 Algorithms

| # | Algorithm | Complexity | Used For |
|---|---|---|---|
| 1 | Topological Sort | O(V+E) | Ordering subjects by prerequisite dependencies; detects cycles |
| 2 | Binary Search | O(N log N) | Minimum number of labs needed to seat all students |
| 3 | Prefix Sum | O(N) | Total lab capacity |
| 4 | DSU (Union-Find) | O(α·N) | Grouping students by subject cohort |
| 5 | Sliding Window | O(N) | Detecting "cheating clusters" — same branch, same lab, same subject in close proximity |
| 6 | Two Pointer | O(N log N) | Detecting overlapping lab time intervals |
| 7 | Tarjan's Bridge-Finding | O(V+E) | Identifying "critical labs" — single points of failure in the dependency graph |
| 8 | Bit Manipulation | O(1)/op | Encoding lab time-slot occupancy as bitmasks |

The program also runs **greedy vs. optimal (round-robin) allocation** side by side, simulates **lab swaps** to see if they reduce cheating clusters, and runs a **stress test** at N = 100 / 1,000 / 10,000 / 50,000 / 100,000 / 200,000 to show how each algorithm scales.

## Output Format (`output.txt`)

A single JSON object containing, among other fields:

- `N, L, S, D` — the input configuration
- `minLabsRequired`, `totalCapacity`, `hasCycle`, `subjectOrder` — core algorithm results
- `labLoads`, `labCapacities`, `labUsagePct` — per-lab occupancy
- `cheatingClusters` — flagged groups (lab, branch, count)
- `criticalLabs` — labs identified as structurally critical
- `greedyAlloc` / `optimalAlloc` — max/min load, wasted capacity %, and cheating-cluster count for each allocation strategy
- `swapSimulations` — before/after cheating and imbalance for each simulated lab swap
- `stressTest` — timing in microseconds for each algorithm at increasing N
- `allocationSteps` — a full per-student assignment log (used to drive the dashboard's animation tab)
- `timings` — microsecond timing for every algorithm on this run

## Dashboard Tabs (`dashboard.html`)

| Tab | Shows |
|---|---|
| **Dashboard** | Headline metrics (students, min labs, subjects, risk clusters, critical labs, capacity, total time, best strategy), lab load bars, risk grid, dependency graph, critical-lab grid, DSU stats, bit-encoding grid |
| **Timings** | Per-algorithm timing cards and bar chart, fastest/slowest algorithm callouts |
| **Greedy vs Optimal** | Side-by-side comparison of the two allocation strategies with a declared winner |
| **Stress Test** | Timing table and chart across the N-ladder (100 → 200,000) to visualize algorithmic complexity |
| **Animation** | Step-by-step playback of `allocationSteps`, with speed control |
| **Heat Map** | Lab-by-lab utilization heatmap, overcrowded/underused lab lists |
| **What If** | Interactive sliders (students, labs, capacity, subjects) to simulate different configurations client-side |
| **Swap Sim** | Results of the simulated lab-pair swaps and whether each was beneficial |
| **Algorithms** | Explainer cards for all 8 algorithms |
| **Complexity** | Big-O reference table |

## How to Run

```bash
# 1. Compile (if needed)
g++ -O2 -o lab_system lab_system.cpp
# (or just run the provided lab_system.exe on Windows)

# 2. Run it and enter/pipe in your data — or reuse input.txt as a reference format

# 3. It writes output.txt in the same folder

# 4. Open dashboard.html in any modern browser

# 5. Drag output.txt onto the drop zone (or click to browse)
```

No installation, server, or dependencies required for the dashboard — it's a single HTML file with everything embedded.
