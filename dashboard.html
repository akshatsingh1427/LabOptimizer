Min Labs · Binary Search · O(N log N)
Why: finding the smallest k labs whose summed capacity ≥ N is monotonic — perfect for binary search.
How: sort caps desc → prefix-sum → bsearch on prefix array for first index where pre[k] ≥ N.
Subjects · Topo Sort · O(V+E)
Why: subjects have prerequisites (DAG); we need a valid linear order and cycle detection in one pass.
How: Kahn's BFS — push 0-indegree nodes, pop & decrement neighbours, repeat.
Risk Clusters · Sliding Window · O(N)
Why: brute pair-compare is O(N²) — we only need to count students sharing (lab, branch).
How: hash students by (labId, branchId); any bucket size ≥ threshold = cluster.
Critical Labs · Tarjan Bridges · O(V+E)
Why: finds labs whose removal disconnects the network. Naive (remove-and-test) is O(V·(V+E)).
How: single DFS tracks disc[u] + low[u]; edge (u,v) is a bridge if low[v] > disc[u].
Capacity · Prefix Sum · O(L)
Why: total seats and per-range capacity must be O(1) lookups during allocation.
How: one linear sweep builds pre[i+1] = pre[i] + cap[i]; queries become subtractions.
Total Time · std::chrono
Why: microsecond precision proves the empirical Big-O on real hardware.
How: wrap each algorithm in high_resolution_clock, sum all duration_cast<μs> deltas.
Best Strategy · Score Compare
Why: picks the allocation with the better composite score so the user sees a clear winner.
How: score = (maxLoad − minLoad) + 10·cheatingClusters — lower wins.

Algorithm Used · Sliding Window + HashMap · O(N)
WhyDetecting clusters of same-branch students in the same lab is a "group counting" problem. Brute pair-compare is O(N²); a single hash pass collapses it to O(N).
HowKey each student by (labId, branchId). Push into a hashmap of vectors. Any bucket whose size ≥ threshold (default 5) is flagged: HIGH ≥ 8, MED ≥ 5, LOW < 5.

TOPO SORT
Execution Order
Load output.txt…
Algorithm Used · Topological Sort (Kahn's BFS) · O(V + E)
WhySubjects form a DAG — S0 must complete before S1 if a prerequisite edge exists. Kahn's BFS produces a valid linear order AND detects cycles for free (if order.size < S → cycle).
How(1) Compute indeg[v] for every node. (2) Push every indeg == 0 node into a queue. (3) Pop u, append to order, decrement indeg[v] for each neighbour; push when it hits 0. Nodes are drawn left-to-right in the resulting order.
Critical Labs
TARJAN
Load output.txt…
Algorithm Used · Tarjan's Bridge-Finding DFS · O(V + E)
WhyA "critical lab" is a bridge in the lab dependency graph — its removal disconnects the network. Naive remove-each-edge-and-BFS is O(V·(V+E)); Tarjan finds every bridge in a single DFS.
HowDFS from any node, store discovery time disc[u] and lowest reachable ancestor low[u]. After visiting child v: low[u] = min(low[u], low[v]). Edge (u,v) is a bridge iff low[v] > disc[u].

Algorithm Used · DSU / Union-Find · ~O(α(N)) ≈ O(1)
WhyTo group students who share a section/lab quickly without rebuilding from scratch each time. BFS-grouping is O(V+E) per query; DSU answers "are A and B in same group?" in near-constant time.
HowEach student starts as own parent. union(a,b) attaches the smaller tree under the larger (union-by-rank). find(x) walks to the root with path compression — flattens the tree on the way up.

Algorithm Used · Bit Manipulation · O(1) per op
WhyEach lab tracks up to 16 (or 64) time slots. A single int is 8 bytes vs 64 bytes for a bool[64]; AND/OR/XOR run in one CPU instruction vs O(K) loops over an array.

Greedy Load Distribution
Load output.txt…
Algorithm Used · Greedy First-Fit Decreasing · O(N · L)
WhyFastest possible heuristic — no priority queue, no extra data structures. Good when you need an answer NOW and "good enough" is acceptable.
How(1) Sort labs by capacity descending. (2) For each student, scan labs left-to-right; drop them in the first lab with space. Result: large labs fill before small ones → uneven loads.
Optimal Load Distribution
Load output.txt…
Algorithm Used · Max-Heap Load Balancing · O(N log L)
WhyTo minimise the maximum load (makespan), the right move at every step is to assign to whichever lab currently has the MOST remaining seats. A max-heap makes that lookup O(log L).
HowPush (remainingCapacity, labId) into a max-heap. Per student: pop top → assign → decrement → push back. Result: loads stay tightly bunched — minimal imbalance.
Why They Differ
GREEDY STRATEGY
Assigns each student to the first available lab with space, sorted by capacity descending. Fast O(N·L) but creates imbalance — large labs fill first.
OPTIMAL STRATEGY
Uses a max-heap priority queue to always assign to the lab with the most remaining capacity. Produces the most balanced distribution — minimizes max load.
Algorithms Compared · First-Fit Decreasing vs Heap-Based LPT
WhyClassic CS trade-off: greedy ≈ 2× faster but ~30% worse balance. Optimal pays one log factor for near-perfect distribution. This panel exposes that trade-off in real numbers.
HowBoth run on the same input. We diff maxLoad, minLoad, wastedPct, and cheatingClusters; the one with the lower composite score gets the WINNER tag.

Why Smart Beats Brute
DSU vs BFS grouping
DSU: O(α·N)≈O(1). BFS: O(V+E) per query. For 200k students: 200k vs billions of ops.
Sliding Window vs brute
Cheating detection: O(N) vs O(N²). At N=200k: 200k vs 40 billion comparisons.
Tarjan vs naive bridges
Tarjan: one DFS = O(V+E). Naive: remove each edge + BFS = O(V·(V+E)). Massively faster.
Bit masks vs bool array
Bool[64]: 64 bytes, O(K) ops. Bitmask: 8 bytes, O(1) ops. 8x memory + instant operations.
